System Architecture

How the layers connect end-to-end

🔁 Orchestrator Pattern

The system uses an Orchestrator → Specialist pattern. A single triage_agent acts as the central brain, receiving every user message. Rather than handing off the conversation, it calls specialist agents as tools — allowing it to chain multiple specialists in one reply (e.g., checking an account balance AND running fraud analysis in the same response).

🌐 Web Layer

A FastAPI server (server.py) exposes a REST API consumed by the single-page frontend (frontend/index.html). The /chat endpoint returns a Server-Sent Events (SSE) stream, enabling real-time token delivery. Static frontend assets are served directly from FastAPI's StaticFiles mount.

🖥️ CLI Layer

main.py provides a standalone terminal interface using the Rich library for pretty-printed output. It connects directly to the same triage_agent without the HTTP layer, useful for local development and testing.

Tool Functions

Atomic capabilities injected into agents

💼 account_tools.py

Simulated banking data layer (replace with real DB calls in production).

  • get_account_balance(account_id) — returns balance, type, status, owner.
  • get_transaction_history(account_id, limit=10) — returns recent transactions with running balance.
  • transfer_funds(from, to, amount, currency) — enforces $50,000 transfer cap; returns transaction ID on success.
  • _verify_credentials(account_id, password) — bcrypt password check with rate-limiting (5 attempts, 60s lockout).

🚨 fraud_tools.py

Heuristic risk scoring engine.

  • analyze_transaction_for_fraud(transaction_id, amount, location, merchant) — adds weighted risk points for high-value amounts (>$10k: +40), suspicious locations (+50), high-risk merchant categories (+30). Returns risk score, level, flags, and recommendation.
  • freeze_account(account_id, reason) — marks account as frozen and logs the action.

🏠 loan_tools.py

Loan eligibility and product catalogue.

  • check_loan_eligibility(customer_id, loan_type, requested_amount) — evaluates credit score and debt-to-income ratio against thresholds per loan type.
  • get_loan_products(loan_type) — returns available product tiers with rates, terms, and max amounts.

🪪 kyc_tools.py

Identity and compliance tooling.

  • verify_customer_identity(customer_id, document_type, document_number) — returns verification status and risk rating.
  • run_aml_check(customer_id, transaction_amount, transaction_type) — flags CTR requirements and suspicious activity; returns AML status and SAR recommendation.

📊 data_synthesis_tools.py

Real-world external data integrations.

  • fetch_stock_data(ticker, period) — pulls OHLCV data from Yahoo Finance for any ticker (JPM, BAC, SPY, etc.).
  • fetch_fred_series(series_id, limit) — pulls any FRED macroeconomic series (FEDFUNDS, MORTGAGE30US, CPIAUCSL, etc.).
  • extract_pdf_text(filename) — quick full-text dump from an uploaded PDF using PyPDF.

📚 rag_tools.py + hybrid_search.py

Intelligent document search pipeline (see RAG section for full details).

  • index_financial_document(filename) — parses, chunks by section, stores vectors.
  • search_financial_documents(query, top_k) — hybrid BM25 + semantic retrieval with Reciprocal Rank Fusion.

API Endpoints

FastAPI routes exposed by server.py

Method Path Auth Description
GET / — Serves frontend/index.html
POST /login — Validates account_id + password (bcrypt). Returns a signed HMAC-SHA256 session token.
POST /logout Token Logs the logout event; client discards token.
GET /me Token Returns the authenticated account_id or 401.
POST /chat Token Streams agent response as SSE. Accepts message, history, and optional custom API key/model.
POST /upload-pdf Token Accepts a PDF multipart upload, saves to uploads/. Sanitises filename to prevent path traversal.
GET /documents Token Lists all PDFs in uploads/ with index status, size, and base-doc flag.
POST /index-document Token Triggers BM25 + vector indexing for a previously uploaded PDF.
POST /shutdown — Gracefully sends SIGINT to the uvicorn process after a 0.5s delay.

Security

Defence layers protecting the system

🔑

HMAC Session Tokens

Login returns a payload.HMAC-SHA256 token. Every protected endpoint verifies the signature server-side. No JWTs, no cookies — stateless and tamper-proof.

🔐

Bcrypt Password Hashing

Passwords are stored as bcrypt hashes with random salts. Comparison uses constant-time checkpw to prevent timing attacks.

🚫

Rate Limiting on Login

After 5 failed attempts, accounts are locked for 60 seconds. Failed attempts for unknown usernames add a 0.5s artificial delay to prevent username enumeration.

🕵️

PII Input Guardrail

Every chat message is scanned by pii_guardrail before reaching any agent. Regex patterns block 16-digit card numbers, SSNs (XXX-XX-XXXX), passport IDs, and long numeric account numbers.

📁

Path Traversal Protection

PDF uploads strip all path components from filenames using pathlib.Path(name).name, preventing directory traversal attacks on the uploads folder.

📋

Structured Audit Logging

Every agent action, login event, guardrail trigger, and error is written to audit.log as a structured JSON entry with UTC timestamp for compliance traceability.

RAG Pipeline

Hybrid search over financial documents

⚙️ How It Works

The RAG (Retrieval-Augmented Generation) pipeline enables the Data Synthesis Specialist to answer precise questions about uploaded PDFs — for example, IRS Publication 560 on retirement plans.

Step 1 — Indexing (index_financial_document)

  • PDF is loaded from uploads/ using the document processor.
  • Text is split into chunks by section, preserving tables as Markdown.
  • Each chunk is embedded into a vector store.
  • A BM25 keyword index is also built over the same chunks.
  • Base documents listed in config.py → BASE_DOCUMENTS are auto-indexed at server startup.

Step 2 — Retrieval (search_financial_documents)

  • The user query is run through both the BM25 keyword index and the vector semantic index in parallel.
  • Results from both are fused using Reciprocal Rank Fusion (RRF) — boosting chunks that rank highly in both methods.
  • Top-k chunks are returned to the agent alongside source document, page number, and section header.
  • The agent synthesises the chunks and cites sources in the final answer.

📄 Base Documents

config.py contains a BASE_DOCUMENTS list. Any PDF filename listed here is automatically indexed when the server starts (if the file exists in uploads/). Currently configured: p560b.pdf — IRS Publication 560 "Retirement Plans for Small Business".

Configuration

LLM models and environment variables

🤖 LLM Model Options (config.py)

Two model roles are defined: TRIAGE_MODEL and SPECIALIST_MODEL. Both use LiteLLM as a provider-agnostic wrapper — simply uncomment the desired provider block:

  • Option A — Groq: llama-3.3-70b-versatile / llama-3.1-8b-instant (free tier, very fast) — requires GROQ_API_KEY
  • Option B — Anthropic Claude: claude-sonnet-4-5 / claude-haiku-3-5 — requires ANTHROPIC_API_KEY
  • Option C — Google Gemini: gemini-2.5-flash (currently active) — requires GEMINI_API_KEY
  • Option D — Mistral AI: mistral-large-latest / mistral-small-latest — requires MISTRAL_API_KEY

🌱 Environment Variables

  • GEMINI_API_KEY (or whichever provider is active)
  • SESSION_SECRET — HMAC signing key for session tokens (change in production)
  • Place all keys in a .env file — loaded automatically via python-dotenv.

🔑 Runtime Model Override

Users can supply their own api_key and model string in the /chat request body. The server builds a temporary LitellmModel and injects it via RunConfig, overriding the server defaults for that request only. This is exposed in the frontend's settings panel.

Account Identity Injection

After authentication, the server decodes the session token to extract the account_id. It then prepends a system message into the conversation with this ID before calling the triage agent. This means no agent ever needs to ask the user for their account number — it is always injected automatically and securely.
